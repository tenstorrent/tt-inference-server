# Dependencies for the LLM_AIPERF venv (Python 3.11): the standard
# `--tools aiperf` LLM benchmark. Ships aiperf plus the base deps the
# orchestrator imports eagerly (aiohttp via
# workflow_module -> test_module.context -> health_tests). Every other
# test runner is lazy-loaded via PEP 562 __getattr__ and is NEVER imported
# on this code path -- so their transitive deps (torchvision, open-clip-torch,
# pillow, scipy, numpy) intentionally do NOT appear below. pip will still
# pull in numpy via transformers / aiperf, but we leave that to pip's
# resolver instead of pinning it ourselves.
#
# The PREFIX_CACHE venv builds on this base via `-r llm-aiperf.txt`.

-c constraints.txt
--extra-index-url https://download.pytorch.org/whl/cpu

# AIPerf benchmarking tool (https://github.com/ai-dynamo/aiperf).
# Our CPU-only torch is pinned to 2.4 so transformers is capped at <5 to
# avoid `ModuleNotFoundError: torch.distributed.tensor.device_mesh` on
# import of `AutoTokenizer`. aiperf itself only requires transformers>=4.56.
torch==2.4.0+cpu
aiperf
transformers>=4.56,<5
pyjwt
requests
datasets

# v2 orchestrator eager-import: workflow_module -> test_module.context ->
# health_tests uses aiohttp at module top.
aiohttp

# Required by models whose HF repo ships a custom tiktoken-based tokenizer
# (e.g. moonshotai/Kimi-*). transformers loads tokenization_kimi.py via
# trust_remote_code, which imports tiktoken (and blobfile to fetch the
# vocab). Without these, AutoTokenizer.from_pretrained fails with an
# ImportError once --tokenizer-trust-remote-code is enabled.
tiktoken
blobfile
